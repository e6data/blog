# Understanding How Iceberg Manages Metadata and Data Files

You can look into the metadata folder above for all the files used here 

Here we’ll walk through a scenario where we first append data to an Iceberg table 2 times and then perform a compaction operation. By examining the metadata files before and after compaction, we’ll gain a concrete understanding of how Iceberg’s table metadata evolves over time.

## Before Compation

### Metadata JSON

Now we will go through how the metadata looks like after two appends. The metadata file usually contains details about the schema of the table, current snapshot id information, and the manifest-list file and other statistics.
This file has the details about the current snapshot and the manifest files associated with it , Apart from the current snapshot, it even stores the history of the snapshot and also the parent-snapshot-id to determine the previous snapshot.

1. **Metadata JSON** – the main JSON file (like `00002-...metadata.json`) which points to the current snapshot of the table.
2. **Manifest List Files** – these are Avro files that list what manifests are part of a given snapshot.
3. **Manifest Files** – Avro files that detail each individual data file added or deleted within that snapshot.

## After Compaction: What Changes?

Compaction in Iceberg is a ***replace*** operation that takes multiple smaller files and rewrites them into fewer, larger files to improve query efficiency.
The goals of data file compaction are:

- Reducing file counts:
1. Reduce the number of data files.
2. Reduce the number of delete files.


- Improving data locality: Apply data clustering to make reads more efficient.

When we run an iceberg compaction operation (Amoro in this case), Iceberg creates a new snapshot and updates the metadata accordingly. This process is crucial for data storage optimization and maintaining optimal data file size.
Compaction is not typically performed over an entire table at a time, as the table is often too large for this. Instead, compaction jobs run periodically such as the last hour, or other specific configurations.
